# Windows 0.1.85：桌面文字清晰度

DeskLink 0.1.85 针对“清晰”画质下的桌面文字发糊做了编码和显示链路优化。

## 调整内容

- 保持 H.264 和 30 FPS 不变；清晰档现在允许最高 2560 × 1440，普通均衡/流畅档仍使用较低上限。
- 将清晰档的默认目标码率从 4 Mbps 提高到 8 Mbps。桌面文字、细线和窗口边缘比摄像头画面更容易暴露量化损失，8 Mbps 为这些高频细节提供了更充足的码率预算。
- 对 Windows Media Foundation H.264 编码器启用清晰档的峰值约束 VBR、CABAC 和较高的质量/速度档，并同步提交平均码率和峰值码率（峰值为平均值的 1.25 倍），允许窗口移动、滚动和文字重绘时短暂使用更多码率；不同厂商编码器不支持这些属性时会安全忽略，不会让会话失败。
- 修正“适应”显示模式：Canvas 重新填满全屏布局盒，再由 `object-fit: contain` 保持原始宽高比，避免全屏时只显示原始 1920 × 1080 画布而留下大面积黑边；“1:1 原始像素”仍保持不放大。
- 保留自动画质的降档策略。网络丢包、控制端解码积压或显示压力持续升高时，主机仍会从清晰逐级切换到均衡/流畅，优先保证可操作性。

## 为什么这一步先于 P2P

公网中继当前只转发端到端加密后的数据，不能读取或修改视频内容。P2P 直连可以降低路径延迟和中继带宽占用，但不会恢复已经被 H.264 量化丢失的像素细节。因此先修正编码质量，才能确认剩余问题是否真的来自网络路径；如果手动双机测试仍显示稳定丢包或高延迟，再增加可选的直连候选探测，不改动现有中继回退路径。

## 手动验收重点

1. 两台 Windows 都升级到 0.1.85，并在控制端选择“清晰”。
2. 在目标电脑上打开包含小字号中文、细边框和滚动列表的窗口，分别观察“适应”和“1:1”。
3. 诊断信息中确认没有持续丢包或解码积压；若网络压力升高，自动模式应自行降档并保持输入可用。
