# Windows 0.1.58：视频参考帧连续性恢复

DeskLink 0.1.58 修复低延迟视频链路中的一个隐藏恢复缺口：一个 H.264 增量帧的数据报分片丢失后，后续增量帧即使完整到达，也可能继续引用已经缺失的参考帧。旧行为可能表现为花屏、短暂停滞或只能等待后续偶然关键帧。

## 本次变化

- 共享 Rust 控制端在视频重组完成后、交给 WebCodecs 之前检查帧号和已确认的分片损失；
- 数据报缺片或完整帧号断层会立即关闭当前增量参考链；
- 引用链不可信时只丢弃视频增量帧，不关闭远程会话，也不影响鼠标、键盘、声音、剪贴板或文件传输；
- 控制端通过已有 Noise 端到端加密控制通道请求新关键帧；
- 如果恢复关键帧再次丢失，只要视频仍在到达，关键帧请求最多每 1 秒重试一次；
- 收到合法关键帧后立即建立新的连续性起点，后续连续增量帧自动恢复交付；
- 新视频配置、画质重配置和屏幕切换会清空旧参考链，迟到的旧流不能恢复新画面。

## 安全与兼容

- 继续使用协议 9，没有新增或修改网络字段；
- relay 仍只转发业务密文，无需升级或重新部署；
- 0.1.57 与 0.1.58 的网络格式兼容，但为了获得一致的恢复行为和发布验证，两台 Windows 电脑仍推荐同时安装 0.1.58；
- 主界面不新增丢包、帧号或重试计数，健康状态保持安静。

## 自动验证

- 纯状态测试覆盖连续帧、帧号断层、已确认分片损失、1 秒重试冷却、配置重置、关键帧恢复和 `u64` 帧号回绕；
- localhost relay + Noise 集成测试覆盖关键帧 10 后跳到增量帧 12：控制端不交付 12，主机收到自动关键帧请求，并从关键帧 13 与增量帧 14 恢复；
- 完整发布门禁继续覆盖 Rust workspace、Clippy、Bun、Python、生产中继协议 9 探针、x64 发布程序和安装器哈希。

真实双机验收由用户完成，重点观察跨网轻微丢包或网络切换后画面能否快速回到清晰实时状态，同时鼠标和其他通道保持连接。
