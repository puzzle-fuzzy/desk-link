# Windows 0.1.60 视频交付自愈与诊断设计

## 背景

0.1.59 已把完整 H.264 帧从无确认的 Tauri Channel 推送改为“一个 WebView 请求在途 + Rust 一个待取帧”的有界交付。该边界解决了旧画面与内存无限积压，但仍有两个发布前隐患：

1. 前端串行拉取在任意一次 `invoke` 拒绝后会静默结束。如果拒绝来自短暂的 WebView/Tauri 调度异常，而不是断线或配置切换，远程画面会永久停在最后一帧，输入和会话却仍显示已连接。
2. 现有诊断能区分网络分片、完整帧、前端收到帧、解码和显示，却看不到 Rust 视频邮箱交付、IPC 溢出和前端拉取失败，人工双机测试仍可能留下无法定位的中间空白。

## 目标与非目标

目标：

- 同一视频流和配置的拉取失败后自动退避重试，成功收到一帧即清空失败级别。
- 停止、断线、重连、切屏或配置切换继续使用 generation + exact key 立即作废旧重试。
- Rust 邮箱累计并输出已交付帧、IPC 增量帧溢出和关键帧替换计数。
- 前端累计拉取失败次数，通过现有渲染指标进入本地脱敏日志。
- 云端诊断能自动区分“网络已有完整帧但 IPC 未交付”“IPC 持续拥塞”和“前端拉取不稳定”。

非目标：

- 不改变协议 9、QUIC/Noise 消息、中继会话或主机编码。
- 不把技术计数展示在主界面，不上传屏幕、输入、设备 ID、密码或完整身份。
- 不修改 macOS，不实现自动更新、代码签名、第二中继节点或语音对讲。
- 不在当前未提交工作区绕过诊断服务的“干净 commit 才能部署”保护。

## 方案比较

### 方案 A：累计指标 + 有界退避重试（采用）

邮箱在已有互斥状态中维护饱和计数，十秒诊断采样读取累计快照；前端串行拉取在失败后按固定上限退避并继续。成功帧重置连续失败次数，生命周期变化使旧等待立即失效。

优点是没有高频日志、没有新网络协议，并能直接定位 Rust→WebView2 边界。代价是客户端新增少量诊断字段，诊断服务必须先接受这些字段再正式分发 0.1.60。

### 方案 B：每次溢出或拉取失败写独立事件

实现简单，但持续卡顿时每秒产生大量重复事件，放大本地日志、上传批次和服务端分析噪声，不采用。

### 方案 C：只扩展本地日志

不会影响服务器兼容性，但无法满足双机远程排查和自动会话分析的目标，不采用。

## 前端自愈

`SerialVideoPull` 保持永远只有一个 `pull` Promise 在途。一次失败后调用可选 `onPullError(error, consecutiveFailures, retryDelayMs)`，等待以下毫秒序列后继续：

```text
100, 250, 500, 1000, 2000, 2000, ...
```

收到一帧后连续失败数归零。`stop()` 或新 `(streamId, configVersion)` 会递增 generation；等待中的定时器可以自然结束，但结束后必须检查 generation，不能发起旧请求或交付旧响应。

控制页面只累计饱和的 `videoPullFailures`，不因单次失败弹窗或重建解码器。已有状态信令负责断线/重连界面；如果只是瞬时 IPC 失败，下一次拉取会自行恢复。

## Rust 邮箱指标

`ControllerVideoMailbox` 新增累计快照：

```rust
pub(crate) struct VideoMailboxMetrics {
    pub(crate) delivered_frames: u64,
    pub(crate) overflow_drops: u64,
    pub(crate) keyframe_replacements: u64,
}
```

- `next` 成功取出一帧后增加 `delivered_frames`；
- 单槽已满且新帧为增量帧时增加 `overflow_drops`；
- 单槽已满且新帧为关键帧并替换旧帧时增加 `keyframe_replacements`；
- 所有计数使用饱和加法；错误键、连续性门主动丢弃和关闭唤醒不伪装成 IPC 溢出；
- 新连接尝试开始时清零指标，配置切换不清零，保证一次连接尝试中的多屏切换可累计；
- `close` 不清零，确保尝试结束时仍能记录最终快照。

现有十秒 `controller_video_metrics` 增加：`delivered_video_frames`、`video_ipc_overflow_drops` 和 `video_ipc_keyframe_replacements`。连接尝试结束时额外记录一次最终累计值，避免十秒内失败的会话完全没有媒体证据。

`controller_render_metrics` 增加 `video_pull_failures`。Rust 边界验证它不超过 `1_000_000`，防止异常 WebView 数值进入日志。

## 云端验证与分析

诊断服务字段白名单增加四个非敏感计数。旧客户端不发送这些字段时按零处理，并通过“是否存在邮箱样本”避免把旧版本误判为 IPC 停滞。

自动分析新增三个结论：

- `video_ipc_stalled`（error）：存在新邮箱样本、网络已完成视频帧，但交付帧仍为零；
- `video_ipc_pressure`（warning）：一次关联会话累计至少 3 次 IPC 增量帧溢出；
- `video_pull_instability`（warning）：前端累计至少 3 次拉取失败。

关键帧替换只作为诊断数字，不单独报警；单次初始配置调度不应制造告警。任何 warning/error 都会阻止 `healthy_video` 结论。

## 发布顺序与兼容

- 产品版本提升为 `0.1.60`，协议仍为 `9`。
- 先提交并部署诊断服务白名单与分析器，再分发 0.1.60 Windows 安装包；否则启用诊断共享的 0.1.60 客户端会让旧服务拒绝包含新增字段的批次。
- 当前阶段可以完成代码、测试和安装包构建，但部署脚本继续要求干净 commit；在用户明确要求 commit/push 前不绕过该保护。
- 诊断默认关闭，因此服务器部署顺序不影响远程控制、视频、输入、音频、剪贴板或文件传输本身。

## 测试

- TypeScript：失败退避序列、成功复位、单请求约束、停止与配置切换取消、交付异常继续。
- Rust 邮箱：三类指标、饱和/重置语义、关闭保留和错误键不计数。
- Windows 诊断：新增字段序列化、前端拉取失败边界验证、最终尝试快照。
- 诊断服务：签名批次接受新增白名单字段；旧字段仍兼容；三个新结论、阈值和健康结论抑制。
- 完整门禁：Rust workspace、Clippy、Python、Bun、TypeScript/Vite、中继协议 9 探针、Windows release 与安装器哈希。
