# Windows 画面新鲜度闭环设计

- 日期：2026-07-20
- 目标版本：DeskLink Windows 0.1.57
- 范围：Windows 控制端 WebCodecs、Tauri 控制端管理器、共享协议和 Windows 主机自动画质；不修改配对、批准、输入、剪贴板、文件、音频或中继服务职责。

## 背景与目标

DeskLink 0.1.54 修复主机 30 FPS 节奏与颜色转换热路径，0.1.56 让新会话默认使用自动画质并保持三档 30 FPS。当前自动画质只观察视频分片丢失：当网络丢包很低、但控制端 WebView2 的硬件解码或 Canvas 绘制跟不上时，主机仍会维持高清档，控制端只能在 `decodeQueueSize > 4` 时静默丢弃单个非关键帧。

H.264 帧之间可能存在引用关系。静默跳过任意增量帧既不能降低主机后续负载，也可能让解码器一直追赶旧画面或等待下一次自然关键帧。0.1.57 的目标是让自动画质同时感知网络与控制端播放压力，并在严重积压时明确回到最新关键帧，而不是通过扩大缓冲伪造流畅。

## 方案比较

### 方案 A：加密播放压力反馈与新鲜度恢复（采用）

控制端每秒汇总 WebCodecs 解码队列峰值和因严重积压执行的新鲜度恢复次数，通过现有 Noise 加密控制通道与网络丢包样本一起发送。主机沿用自动画质迟滞，根据网络或播放压力逐级降档；控制端严重积压时清空解码队列、保留最后已显示画面并请求新关键帧。

优点是能解决低丢包但控制端处理不过来的真实盲区，并让主机实际降低码率或分辨率。代价是控制消息结构变化，需要协议版本从 8 升至 9，两台电脑必须同步升级。

### 方案 B：只在控制端重启解码器

不改变协议，队列过深时重建 WebCodecs 解码器并请求关键帧。实现较小，但主机仍持续发送原档位，性能不足时会周期性重启，画面停顿更明显。

### 方案 C：自动模式固定从 720p 开始

不采集播放压力，直接牺牲清晰度换取更低负载。实现最简单，但性能良好的电脑也无法默认获得清晰画面，并不是真正的自适应。

## 架构与数据流

```text
WebCodecs 接收帧
  ├─ 记录本秒 decodeQueueSize 峰值
  ├─ 正常压力：继续提交解码
  └─ 严重且持续积压：关闭旧解码器 → 重建同档解码器 → 请求关键帧
           │
           ▼ 每秒一次本地 IPC
Tauri ControllerManager 合并当前 stream_id 的有界播放压力样本
           │
           ▼ 与分片收包/缺失增量一起取出
Noise 加密 ControlMessage::VideoNetworkFeedback
           │
           ▼
Windows 主机 AdaptiveVideoQuality
  ├─ 持续网络丢包或播放压力：清晰 → 均衡 → 流畅
  └─ 持续健康且无播放压力：流畅 → 均衡 → 清晰
```

中继只看到已有控制通道的密文，不解析、保存或聚合播放压力。协议版本升级只用于防止新旧客户端误解控制消息，现有中继目录已经按两端提交的版本做相等匹配，因此服务器二进制无需升级。

## 控制端播放压力

新增纯 TypeScript 状态模块，负责以下行为：

- 每个视频流独立记录解码队列峰值；流切换和断线必须重置；
- 队列深度 `0–2` 为正常，继续提交；
- 队列深度达到 `5` 后计为一次严重压力；连续三次严重压力才触发新鲜度恢复，避免单次突发造成重启；
- 两次恢复至少间隔 5 秒；冷却期间继续记录峰值但不重复重建；
- 恢复操作关闭待处理 `VideoFrame` 和旧 `VideoDecoder`，保留 Canvas 上最后一帧，使用当前硬件/软件偏好重新创建解码器，并通过现有命令请求关键帧；
- 不再静默跳过单个 H.264 增量帧。

每秒上报一次有界样本：`streamId`、`peakDecodeQueueSize` 和 `freshnessRecoveries`。前端不发送累计设备或会话身份，不在界面展示技术数字。

## 协议与主机策略

`ControlMessage::VideoNetworkFeedback` 增加：

- `decode_queue_peak: u16`
- `freshness_recoveries: u16`

字段只表示最近一个反馈周期。Tauri 边界拒绝错误流 ID，将队列深度限制为 `0..=64`、恢复次数限制为 `0..=16`，每次 worker 周期取出后清零；WebView 即使异常也不能制造无界计数。

自动画质把以下条件纳入退化判断：

- 至少 40 个分片且丢失率不低于 5%；
- 解码队列峰值不低于 5；
- 本周期发生至少一次新鲜度恢复。

新鲜度恢复已经由三个连续严重队列观察触发，因此它是强退化信号：不在冷却期时立即逐级降低一档。仅有队列峰值或网络丢包时仍沿用连续 2 个退化样本，切换后保留现有 8 个反馈样本冷却。健康升级仍需至少 40 个分片、丢失率不高于 0.5%、队列峰值不高于 2 且没有恢复，并沿用连续 12 个健康样本。手动档完全忽略自动反馈。

## 错误与恢复边界

- 播放压力 IPC 失败只跳过当前样本，不中断画面、输入或连接；
- 网络反馈发送失败继续沿用当前连接错误处理，因为它表明加密控制通道已经不可用；
- 解码器新鲜度恢复与现有“硬件解码失败后切软件”分离，不把主动清理积压记为解码错误；
- 画质切换、显示器切换、配置版本变化、断线和重新连接会清空旧流压力，迟到样本不能影响新流；
- 最低“流畅”档仍过载时只执行有冷却的新鲜度恢复，不降低到 30 FPS 以下，也不增加播放缓冲。

## 测试与验收

- Bun 单元测试覆盖峰值合并、三次严重压力触发、5 秒冷却、健康样本重置、流切换重置和每秒样本取出；
- 协议 round-trip 测试覆盖新增反馈字段，并确认协议版本为 9；
- Rust 主机单元测试覆盖仅播放压力降档、健康恢复需要低队列、手动档忽略压力和稀疏网络样本仍能处理严重播放压力；
- Tauri 控制端测试覆盖错误流拒绝、有界字段、合并峰值、恢复次数饱和和取出清零；
- 现有 Bun、Python、Rust workspace、Clippy、生产中继探针与 Windows 安装器门禁全部通过；
- 人工双机验收重点观察远端视频播放、窗口拖动和快速滚动：画面不应长期落后操作，严重积压恢复时 Canvas 保留最后一帧，不显示黑屏，两台电脑必须使用 0.1.57。

## 明确不做

- 不引入 RTT 探测、带宽估算器、TCP/TLS 备用通道或新中继服务；
- 不把队列深度、帧数或丢包数字重新放回主工具栏；
- 不自动把手动画质改回自动；
- 不改变文件、剪贴板、音频或远程输入的队列优先级；
- 不在本阶段处理代码签名、第二中继节点或自动安装。
