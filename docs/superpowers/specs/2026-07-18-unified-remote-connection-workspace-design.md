# DeskLink 统一远程连接工作台设计规格

日期：2026-07-18  
状态：已获用户确认，待规格评审  
适用端：macOS、Windows、iOS  
本轮实现范围：macOS 页面与 macOS 远程内核验证；Windows 页面同步；iOS 仅建立统一页面规范，不新增 iOS 工程

## 1. 背景与问题

DeskLink 的核心任务不是查看本机运行状态，而是完成两类远程桌面动作：

1. 控制另一台设备。
2. 允许另一台设备连接本机。

现有 macOS 和 Windows 页面把“本机状态”放在首页首要位置，并直接展示视频包、输入事件、关键帧等运行指标。这些信息对诊断有价值，但不是普通用户启动应用后的首要任务，会让产品看起来像本机服务监控器，而不是远程连接工具。

iOS 当前没有可编辑的客户端工程，因此本规格先定义 iOS 的控制端页面结构、文案和状态行为，作为未来实现的统一基线。iOS 仍然是控制端，不扩展为完整被控端。

## 2. 设计目标

- 启动后立即让用户理解“如何连接设备”。
- 让“控制另一台设备”和“允许别人连接”都可发现，但明确主次。
- 将本机权限、诊断、运行指标和已批准设备降为二级信息。
- macOS、Windows、iOS 使用同一套任务名称、状态词和操作优先级。
- 不改变 Windows 远程内核，不因为页面同步引入跨平台运行时改动。
- 保持活动远程会话专注于被控画面和必要会话操作。

## 3. 非目标

- 本轮不实现 Windows 远程内核，也不在 macOS 上模拟或调用 Windows 系统能力。
- 本轮不新增 iOS Xcode 工程、网络协议或远程桌面内核。
- 不删除诊断和本机状态数据；只改变它们的展示层级和入口。
- 不改变现有连接码、审批、权限检查、重连和关键帧请求协议。

## 4. 统一信息架构

### 桌面端（macOS / Windows）

主导航由原来的四个并列入口调整为以下层级：

| 层级 | 入口 | 目的 |
| --- | --- | --- |
| 主入口 | 连接设备 | 输入或粘贴连接码，连接到另一台设备 |
| 主入口内 | 最近设备 | 快速重连已保存的远程设备 |
| 次入口 | 共享此设备 | 生成连接码，让别人连接本机 |
| 次入口 | 已批准设备 | 管理可自动连接本机的设备 |
| 低频入口 | 设置 / 诊断 | 查看权限、运行状态、密钥和诊断详情 |

不再把“本机状态”作为首页主导航标签。若保留桌面端的侧栏或导航栏，标签应使用“连接设备”“共享此设备”“已批准设备”“设置”，其中设置承载诊断。

### iOS 控制端

iOS 使用底部导航或顶部返回结构表达同一套任务层级：

- 连接设备：默认页，输入连接码和查看最近设备。
- 已保存设备：查看和重连设备。
- 更多：查看连接偏好、诊断和帮助。

iOS 不展示“共享此设备”作为可执行的被控端入口。若未来需要展示跨设备共享能力，只能以说明卡片或帮助入口表达，不得暗示 iOS 当前支持被控。

## 5. 默认首页：连接设备

默认启动页为“连接设备”，而不是本机运行状态。首次启动和后续启动都遵循以下规则：

- 默认聚焦“控制另一台设备”。
- 记录用户上一次成功使用的角色；若用户连续使用共享流程，下一次可在保留主入口的同时提高“共享此设备”卡片的视觉权重，但不得替换默认连接入口。
- 如果存在活动远程会话，直接恢复会话页，不进入首页。
- 如果有待审批的连接请求，首页显示轻量提醒，并立即提供审批弹窗入口；不得用运行指标替代提醒。

首页内容顺序：

1. 页面标题“连接设备”和一句解释“输入连接码，开始控制另一台设备”。
2. 粘贴连接码主操作；手动输入连接码作为同级备用操作。
3. 最近设备列表，每个设备提供连接操作和必要的最近连接信息。
4. “允许别人连接此设备”次要卡片，提供生成连接码入口。
5. 右上角本机可连接状态，以及设置/诊断入口。

首页不直接展示以下内容：视频数据包、输入事件、关键帧请求数量、stream ID、密钥材料和完整本机权限明细。

## 6. 本机状态的降级展示

本机状态只保留一行可行动的摘要：

- “本机可被连接”：权限和主机服务已准备好。
- “需要处理”：缺少屏幕录制或辅助功能权限。
- “未开启共享”：主机服务尚未启动。
- “连接中”：正在等待审批或建立主机连接。

摘要位于桌面端右上角，iOS 不展示本机被控状态。点击桌面端摘要后打开设置/诊断抽屉，内容按以下顺序组织：

1. 权限检查和修复入口。
2. 当前共享连接状态。
3. 待审批请求。
4. 已批准设备管理。
5. 运行指标和错误详情。

运行指标默认折叠，只有诊断场景展开。状态摘要必须能够把用户带到下一步，而不是只显示一个不可操作的颜色或数字。

## 7. 共享此设备流程

点击“生成连接码”后：

1. 检查必要系统权限。
2. 缺少权限时，在上下文内明确告诉用户需要开启什么，并提供前往系统设置的入口。
3. 权限具备时启动主机配对，显示连接码、有效性或等待状态，以及复制操作。
4. 收到控制请求时使用高优先级确认弹窗，显示请求设备名称、平台、请求时间和允许/拒绝按钮。
5. 审批完成后回到共享状态摘要；不把主机运行指标推到首页。

用户主动停止共享、拒绝请求或撤销设备时，使用短暂 Toast/状态摘要反馈结果。

## 8. 控制另一台设备流程

控制端的连接入口支持：

- 粘贴连接码：默认主按钮。
- 手动输入连接码：同级辅助按钮。
- 最近设备：直接连接，失败时回退到连接码流程。

连接状态统一使用以下文案：

| 状态 | 用户可见文案 | 页面动作 |
| --- | --- | --- |
| idle | 准备连接 | 输入或粘贴连接码 |
| pairing | 等待确认 | 取消连接 |
| connecting | 连接中 | 取消连接 |
| connected | 已连接 | 进入远程会话 |
| reconnecting / recovering | 正在恢复连接 | 等待、取消或断开 |
| frozen | 画面暂时冻结 | 请求关键帧、断开 |
| failed | 连接失败 | 查看原因、重新连接 |

连接失败和权限问题使用可理解的下一步说明，不默认展示内部错误码或 stream ID。

## 9. 活动远程会话

活动会话保持沉浸式远程画面，视觉优先级如下：

1. 远程画面。
2. 当前连接状态和目标设备名称。
3. 断开连接、请求关键帧等必要操作。
4. 诊断指标和网络细节放入工具栏的更多菜单或弹出面板。

本机运行状态不进入会话页。若发生重连、冻结或失败，使用会话内横幅说明影响和下一步；不跳转到本机状态页。

## 10. 跨平台视觉与交互约束

- 统一使用“连接设备”“共享此设备”“已保存设备”“设置 / 诊断”等任务语言。
- 统一主次关系：连接设备是主动作，共享本机是次动作，诊断是低频动作。
- 桌面端允许使用侧栏、顶部栏和弹出抽屉；iOS 使用导航栏、底部导航、页面卡片和 sheet。
- 颜色只表达状态和动作，不用大面积状态色渲染本机监控感。
- 主页面信息密度保持轻量，技术指标只在诊断上下文出现。
- 所有高风险动作（允许控制、停止共享、撤销设备、断开会话）必须有明确动作对象和反馈。
- 触控端控件需要满足 iOS 可点击区域要求；桌面端键盘焦点和粘贴操作必须可用。

## 11. 实施拆分

### 页面层

- macOS：将默认入口从 HostHomeView 的 overview 调整为连接设备工作台；重排 ControllerHomeView、HostHomeView、DeskLinkShell 和 DiagnosticsView 的入口层级。
- Windows：同步导航标签、首页布局、连接/共享主次关系和状态文案；保持 controller/host 运行时接口不变。
- iOS：把本规格作为未来控制端页面契约；当前仓库无 iOS 工程，不虚构实现或验收结果。

### 内核层

- 本轮只在 macOS 上修改或验证必要的连接流程适配。
- Windows 内核不修改。
- 不改变现有邀请、审批、连接、重连、关键帧请求和权限检测协议。

## 12. 验收标准

- macOS 和 Windows 启动后第一视觉焦点是“连接设备”，不是本机运行状态。
- 首页可以在一次操作内粘贴连接码并开始连接。
- 共享本机仍能从首页发现，并在需要时生成连接码。
- 本机运行指标不再占据首页主卡片；可从设置/诊断进入。
- 待审批请求在首页和系统/应用层面都有清晰入口，不会被隐藏在诊断页。
- macOS、Windows 的任务名称、状态文案和操作顺序一致。
- iOS 未来页面可按同一信息架构实现，且不会误导用户认为 iOS 支持被控。
- macOS 既有连接、审批、重连、冻结和断开行为回归通过。
