# Windows 0.1.58 视频参考帧连续性恢复设计

## 背景与问题

DeskLink 0.1.57 已经限制主机编码队列、只绘制控制端最新解码帧，并在 WebCodecs 解码队列持续积压时重建解码器。该闭环解决的是“帧完整到达，但本机播放跟不上”。

当前仍存在另一条独立风险：视频使用低延迟 QUIC 数据报传输，一个 H.264 增量帧只要缺少任一分片，重组器就无法生成该帧。控制端虽然记录丢包并降低自动画质，却仍可能把后续完整到达的增量帧交给解码器。后续增量帧可能引用已经丢失的参考帧，因此会出现花屏、短暂停滞或一直等待偶然关键帧的情况。

## 目标

- 数据报缺片或完整帧编号断层后，不再向解码器交付引用链不完整的增量帧。
- 立即通过现有 Noise 加密控制通道请求新的 H.264 关键帧。
- 如果请求产生的关键帧也丢失，只要后续视频仍在到达，最多每 1 秒自动重试一次，而不是永久等待。
- 收到合法关键帧后自动恢复正常增量帧交付。
- 不改变协议 9、不修改 relay、不增加用户设置或主界面技术信息。

## 方案比较

### 方案 A：控制端参考帧连续性门（采用）

共享 Rust 控制端在视频重组完成后、向平台 UI 交付前检查分片损失、帧号连续性和关键帧状态。发现引用链不可信时丢弃后续增量帧并有界请求关键帧。

优点是恢复只在真实异常后发生，延迟低，不增加健康网络的常态码率，并同时保护 Windows Tauri 与共享 FFI 使用者。代价是需要明确管理流切换、配置切换和请求重试状态。

### 方案 B：主机周期性强制关键帧

每隔固定时间编码 IDR 帧，使丢失引用链最终自然恢复。实现较简单，但关键帧会制造持续带宽和编码尖峰，恢复时间取决于周期，并不能阻止恢复前继续向解码器提交无效增量帧。

### 方案 C：视频改为可靠有序流

数据完整性最高，但任何丢包都会阻塞后续画面，远程桌面会从“偶发丢帧”退化成“画面越来越旧”，不符合低延迟交互目标。

## 架构

新增 `crates/desklink-ffi/src/video_continuity.rs`，只负责纯状态决策：

- 记录最后成功交付的帧号；
- 记录当前是否必须等待关键帧；
- 记录最近一次自动关键帧请求时间；
- 接收“已确认分片损失”和“当前完整帧”观察；
- 返回 `Present`、`Drop` 或 `DropAndRequestKeyframe`。

`ControllerRuntime::next_event` 仍负责网络、解密和重组。它把重组器报告的缺片通知连续性门，再把完整帧交给连续性门决策。只有 `Present` 才能进入 `FrameAssembler::accept_for_present` 并产生 `ControllerEvent::H264AccessUnit`；`DropAndRequestKeyframe` 会通过已有 `ControlMessage::RequestKeyframe` 发送请求，然后继续接收会话数据。

前端、Windows 主机、relay 和协议结构保持不变。

## 状态规则

1. 新视频配置到达时清空旧帧号并进入“等待关键帧”。
2. 等待期间收到增量帧：丢弃；若从未请求或距上次自动请求至少 1 秒，同时请求关键帧。
3. 正常期间发现重组器丢弃缺失分片：立即进入“等待关键帧”。
4. 正常期间发现完整帧号不是上一交付帧号加一：进入“等待关键帧”，丢弃当前增量帧并按冷却规则请求关键帧。
5. 任意等待状态收到关键帧：立即交付，将其作为新的连续性起点并清除请求冷却。
6. 旧帧、重复帧和错误流仍由现有重组器规则拒绝，不能改变新流状态。
7. 帧号加一使用 `wrapping_add(1)`，允许 `u64` 帧号自然回绕。

## 错误处理与安全边界

- 自动请求沿用现有 Noise 端到端加密控制通道，中继不能读取或伪造业务请求。
- 请求冷却固定为 1 秒，防止持续丢包时形成控制消息风暴。
- 连续性恢复只丢弃视频增量帧，不关闭会话，不影响鼠标、键盘、声音、剪贴板或文件传输。
- 自动画质仍使用现有丢包和播放压力反馈；本阶段不增加新的协议字段。
- 新配置、屏幕切换和画质重配置都必须重置连续性，旧配置的帧不能恢复新解码链。

## 测试

- 纯状态单元测试覆盖：首个关键帧、连续增量帧、帧号断层、已知分片损失、1 秒请求冷却、关键帧恢复、配置重置和 `u64` 回绕。
- 共享控制端集成测试覆盖：关键帧之后跳过一个帧号时不交付断层增量帧、主机收到自动关键帧请求、随后关键帧恢复交付。
- 回归验证覆盖 `desklink-ffi`、完整 Rust workspace、Clippy、Bun、Python 门禁、Windows 发布构建和安装器清单。
