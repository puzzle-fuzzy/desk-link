# Windows 视频通道评估：4K、H.264 High Profile 与 P2P

评估基于当前 0.1.89 代码和协议，不把未验证的实验能力伪装成稳定功能。

## 结论先行

| 方向 | 对清晰度的直接收益 | 当前风险 | 决策 |
| --- | --- | --- | --- |
| 4K 原生编码 | 高；可减少 4K 屏幕被压到 2560 × 1440 后的缩放损失 | 码率、CPU、内存和视频 Datagram 丢包概率同时上升 | 暂不默认开启；先做局域网/低丢包实验档 |
| H.264 High Profile | 中等；同等画质下可提高压缩效率，不能恢复已丢失的分辨率 | 部分 WebView2/硬件解码器不接受 High，需要探测和回退 | 最适合先做，必须保留 Main Profile 回退 |
| P2P 视频通道 | 间接收益；降低 RTT 和中继带宽，不会修复编码量化造成的模糊 | 需要 STUN/ICE、候选交换、身份绑定、NAT 失败回退和新的诊断面 | 不作为本轮清晰度修复；作为后续可选传输层 |

## 1. 当前 4K 硬约束

当前协议、能力协商和 Windows 编码器都把最大画面限制为 `2560 × 1440`：

- `crates/desklink-protocol/src/lib.rs`：`MAX_MVP_WIDTH/HEIGHT`；
- `apps/windows/src/encoder.rs`：`MAX_ENCODE_WIDTH/HEIGHT`；
- `DeviceCapabilities` 和 VideoConfig 校验会拒绝更大的尺寸；
- 视频 Datagram 每包有效载荷只有 1024 字节，单帧最多 `4096` 个分片，即约 4 MiB。

因此，直接把两个常量改成 3840 × 2160 并不能得到安全的 4K：协议边界、能力协商、重组内存和实际编码压力必须同时升级。

## 2. 4K 的量化成本

当前清晰档是 2560 × 1440、18 Mbps、30 FPS。按相同的每像素码率，4K 需要约 40.5 Mbps；单帧 NV12 原始缓冲约 11.87 MiB，而当前 2560 × 1440 约 5.27 MiB。

在 30 FPS、每片 1024 字节的情况下，平均帧的理论分片和完整到达概率如下：

| 配置 | 平均帧大小 | 分片数 | 独立 1% 丢包时整帧到达 | 独立 3% 丢包时整帧到达 |
| --- | ---: | ---: | ---: | ---: |
| 2560 × 1440 / 18 Mbps | 73.2 KiB | 74 | 47.5% | 10.5% |
| 3840 × 2160 / 30 Mbps | 122.1 KiB | 123 | 29.0% | 2.36% |
| 3840 × 2160 / 40.5 Mbps | 164.8 KiB | 165 | 19.0% | 0.66% |

这是独立丢包的保守估算，真实网络可能有突发丢包，结果会更差。当前链路已经具备缺片后请求关键帧的恢复机制，但 4K 会让“完整帧被丢弃”的频率明显增加，表现为画面停顿而不是单纯变模糊。

## 3. H.264 High Profile

当前 Windows Media Foundation 输出类型明确设置为 Main Profile；控制端已经根据 SPS 读取 `profile_idc` 生成 WebCodecs `avc1.xxxxxx`，所以协议层不需要新增一种 Codec 名称。

High Profile 的合理接入方式是：

1. 控制端在收到 SPS 后调用 `VideoDecoder.isConfigSupported()` 探测 `avc1.640...`；
2. 探测成功时，主机使用 High Profile；
3. 探测失败、解码器报错或首帧超时，回退到 Main Profile 并请求关键帧；
4. 诊断中记录“High 已启用/回退 Main”，但不把内部 Profile 名称塞进主界面。

High Profile 只改善压缩效率和边缘保持，不会把 2560 × 1440 变成 4K，也不能替代高质量缩放和足够码率。它应先于 4K 和 P2P 落地，因为改动范围较小且可以无感回退。

## 4. P2P 视频通道

当前 `QuicClient` 只创建到公网中继的 QUIC Endpoint：

- 客户端绑定 `0.0.0.0:0`；
- `QuicClient` 内部 Endpoint 不对上层暴露；
- 中继只做会话匹配和密文转发；
- 代码中没有 STUN、ICE、候选地址交换或直连 QUIC 建链。

因此 P2P 不是“把 relay 地址替换成对方地址”这么简单。完整方案至少需要：

- STUN 发现公网候选地址；
- 通过现有加密控制通道交换候选并完成 ICE 打洞；
- 直连后的 Noise 身份和会话 transcript 绑定，防止候选劫持；
- NAT 失败、对称 NAT、企业网络和 UDP 被阻止时自动回到 relay；
- 直连/中继两条链路的独立指标和用户可理解的状态文案。

P2P 能降低延迟和中继流量，但不会恢复已经被 H.264 丢掉的像素细节，所以不应作为本轮清晰度问题的第一修复项。

## 推荐实施顺序

1. **先做 High Profile 能力探测与 Main 回退**：不改变默认中继，不改变用户连接流程。
2. **增加 4K 局域网实验档**：只在两端声明支持、RTT/丢包满足阈值且不经过公网中继时开放；需要新的帧分片/恢复预算和明确的实验开关。
3. **抽象传输层并加入 ICE/P2P**：直连只承担视频/音频 Datagram，控制、审批、文件和剪贴板仍保留可靠加密通道；任何失败都回到现有 relay。
4. **最后再评估公网 4K**：在没有可靠帧恢复或 FEC 前，不把公网 4K 作为默认能力。

## 当前第二阶段进展

协议层已加入离线 `VideoFrameBudget` 测量类型和 4K 实验尺寸常量。它只计算平均编码帧大小、NV12 原始帧内存和当前 Datagram 分片数，不会改变默认的 `2560 × 1440` 校验上限，也不会让公网中继自动进入 4K。后续局域网实验必须先把路径类型、丢包、RTT 和帧恢复策略接入能力门控，再开放实际编码档位。
