# Windows 0.1.57：画面新鲜度闭环

DeskLink 0.1.57 解决“网络丢包很低，但控制端解码或显示跟不上时画面仍然越积越慢”的盲区。自动画质现在同时观察网络传输和控制端播放压力，不增加视频缓冲，也不把技术指标放回主工具栏。

## 本次变化

- 控制端每秒汇总当前视频流的 WebCodecs 解码队列峰值；
- 播放压力通过已有 Noise 端到端加密控制通道发送，中继无法读取；
- 持续解码积压与网络丢包共用原有自动画质迟滞，主机逐级降低码率或分辨率；
- 连续三个严重积压观察会触发本地新鲜度恢复：关闭旧解码队列、保留 Canvas 最后一帧，并请求新的 H.264 关键帧；
- 删除此前在队列过深时静默跳过单个 H.264 增量帧的行为，避免后续帧继续引用已丢弃内容；
- 新鲜度恢复与真正的解码器错误分开计数，不会误触发硬件解码到软件解码的降级；
- 流切换、画质切换、断线和重连会清空旧流样本，迟到反馈不能影响当前会话。

## 安全与兼容

- WebView 上报的队列峰值限制为 `0..=64`，单周期恢复次数限制为 `0..=16`；Tauri 只接受当前视频流并在取出后清零；
- 手动“流畅、均衡、清晰”档忽略自动压力反馈；
- 流畅、均衡和清晰档仍保持 30 FPS，不使用累积旧帧换取表面连续；
- 控制消息结构升级为协议 9，两台 Windows 电脑必须都安装 0.1.57；旧版会在目录查询阶段得到明确的版本不兼容提示；
- 中继只比较双方登记的协议版本并转发密文，现有公网中继无需重新部署。

## 自动验证

- Bun 单元测试覆盖三次严重积压、健康打断、5 秒恢复冷却、样本取出、数值边界与流重置；
- Rust 协议测试覆盖协议 9 反馈往返；
- Tauri 测试覆盖当前流合并、峰值选择、恢复次数饱和、迟到流忽略、错误值拒绝和取出清零；
- Windows 主机测试覆盖播放压力立即/迟滞降档、低队列健康恢复和手动档隔离；
- 完整发布门禁继续覆盖 Bun、Python、Rust workspace、Clippy、生产中继探针、x64 发布二进制与安装器哈希。

真实双机验收由用户完成，重点观察远端视频播放、快速滚动和窗口拖动时，画面是否保持接近当前操作而不是持续落后。
