# Windows 0.1.42：可信正式版本检查

## 用户结果

- DeskLink 会在后台轻量检查 GitHub 仓库的最新稳定 Release，但不会打断或重绘正在进行的远控会话；
- “设置 / 诊断”中的“应用更新”会区分正在检查、已经最新、发现新版本、尚无正式版本、发布不完整和网络失败；
- 只有版本号严格符合稳定 SemVer、包含匹配版本的 Windows x64 安装器，并且安装器清单明确通过签名门禁时，才会出现“查看并安装”；
- 升级入口始终在系统默认浏览器中打开固定的 DeskLink GitHub Release 页面，远程返回值不能改变跳转目标；
- 检查失败不影响主机在线、控制会话、文件传输或系统声音，用户可以在设置页手动重试。

## 发布安全边界

GitHub 的标签发布工作流现在拥有单独的正式发布门禁。创建 Release 前必须同时满足：

1. 标签与五处 Windows 版本号一致；
2. 应用和安装器已通过现有 Authenticode 签名策略；
3. 安装器文件名、长度和 SHA-256 与生成清单完全一致；
4. Windows Release 验证报告的版本与结果一致；
5. 清单中的 `signed` 与 `passed` 均为 `true`。

任何条件不满足时，工作流会在创建 GitHub Release 之前失败。普通分支 CI 仍可构建未签名的本地测试安装器，但不会把它发布给客户端。

## 为什么暂不静默更新

DeskLink 使用自定义的当前用户安装器，并且正式 Authenticode 身份尚未配置。0.1.42 因此只提供可信版本发现和用户主动升级入口，不会在后台下载或执行可执行文件。后续自动安装必须增加独立的更新签名密钥、可恢复下载和安装失败回滚，不能仅依赖 HTTPS 或发布清单中的哈希。

## 自动验证

- Bun 测试覆盖版本解析、数字比较、草稿/预发布拒绝、下载域名限制、安装器资产完整性、已签名清单校验、无 Release 与当前已最新状态；
- Python 测试覆盖正式发布标签、签名状态、安装器长度与 SHA-256，以及仓库名称输入校验；
- TypeScript/Vite、Rust 格式、Clippy、workspace 测试和 Windows x64 安装器门禁继续作为发布前验证。
