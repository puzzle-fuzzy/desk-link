# Windows 0.1.54：远程画面流畅度优化

0.1.54 针对控制端持续画面略卡的问题优化主机视频热路径。现有脱敏日志显示约 5,700 个视频包只丢失 7 个，WebCodecs 的接收、提交和显示始终只相差约 1 帧且没有解码恢复，但完成画面只有约 16–18 FPS，因此瓶颈不在中继或控制端 Canvas。

## 帧节奏修复

旧逻辑用上一张已编码帧的时间戳进行严格间隔比较。60 Hz 采集常产生约 `16,666 µs` 的间隔，两帧合计 `33,332 µs`，会比 30 FPS 的整数阈值 `33,333 µs` 少 1 微秒，导致本应编码的第二帧被跳过，实际节奏容易退化到约 20 FPS。

新节奏器：

- 维护绝对的下一帧截止时间，而不是从最近一次实际提交重新漂移；
- 允许 2 ms 的采集时间戳抖动与微秒取整误差；
- 迟到时直接跨过已经错过的周期，不补发陈旧帧；
- 切换屏幕、切换画质、强制关键帧或系统时钟回退时安全重置；
- 自动测试确认 60 个 `16,666 µs` 采样在 30 FPS 配置下稳定选择 30 帧，而不是约 20 帧。

## 颜色转换热路径

原 BGRA/RGBA→NV12 转换在每个目标像素上执行横纵缩放除法，并在生成 Y 与 UV 时重复读取源像素。新实现预先计算目标列和目标行的源偏移，再以 2×2 块一次读取四个像素，同时生成四个 Y 和一组 UV。

- 算法输出与旧参考实现在缩放、行距和两种像素顺序下逐字节一致；
- 本机 release 模式 1920×1080 测量：优化路径约 4.40 ms，旧参考路径约 7.42 ms，提升约 1.69 倍；
- 真实 Windows DXGI 捕获与 Media Foundation H.264 编码烟雾测试通过。

## 边界

本阶段不提高默认 4 Mbit/s 码率、不增加视频排队深度，也不改变丢帧优先的低延迟策略。DeskLink 网络协议仍为版本 8，公网中继无需重新部署。最终体感仍需在两台 Windows 电脑上比较 0.1.53 与 0.1.54 的滚动、窗口拖动和视频播放场景。
