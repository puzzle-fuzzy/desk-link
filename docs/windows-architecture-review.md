# DeskLink Windows 架构复盘

- 日期：2026-07-20
- 范围：Windows 10/11 x64 桌面端、共享 Rust 核心、QUIC relay、安装器与发布门禁。

## 结论

DeskLink 的安全会话与 Windows 媒体链路已经具备可靠的 MVP 基础，主要问题不在“Rust/Tauri 是否能用”，而在发布形态与产品默认路径曾经同时存在两套架构：现代 Tauri 应用旁仍打包旧辅助 host，公网中继旁仍保留内置局域网中继，源码默认值、文档和已构建产物也发生过漂移。

本轮将 Windows 产品收敛为一个入口、一套网络模型和一条发布链：

```text
DeskLink.exe
├─ WebView2 UI：中文界面、WebCodecs、受限输入采集
└─ Rust/Win32 安全边界
   ├─ DPAPI 身份 / 配置 / 信任 / 传输恢复
   ├─ 本地原生批准与撤销
   ├─ QUIC + 系统 TLS 校验
   ├─ Noise 双向认证与业务 AEAD
   ├─ DXGI → Media Foundation H.264
   └─ 有界 SendInput
          │
          ▼
公网或自建 relay：只匹配会话并转发业务密文
```

## 已解决的问题

### 网络与安全

- 删除桌面进程内的临时自签名局域网 relay，以及跳过服务器证书身份校验的客户端路径。
- 默认使用部署中的公网 relay，并通过系统信任链校验 `turn.p2p.yxswy.com`。
- 首次运行由 Rust 生成会话 ID 与 relay 凭据，DPAPI 加密保存；WebView 不接触已保存密钥。
- 删除旧版连接码入口与 loopback 配置迁移，未发布版本统一使用设备 ID、访问密码和公网中继目录。
- 普通常驻主机在单轮重试预算耗尽后会按最大退避继续下一轮恢复，不会因短时会话占用永久停机；一次性配对仍严格受邀请过期时间约束。
- relay 目录登记和查询携带应用协议版本；访问密码匹配后若版本不兼容，会在建立会话前返回不可重试结果，并依据版本大小提示升级控制端或目标电脑。旧目录信封继续可解码，错误密码不会泄露设备在线状态或版本。
- 协议层拒绝 F13+、NUL/控制字符、越界指针/滚轮和未知修饰键，不把合法性只留给最终 `SendInput`。

### 发布架构

- 安装器只部署 `DeskLink.exe`；程序目录不再包含第二个 host 入口，避免两个生命周期模型并存。
- Tauri 发布构建固定启用 `custom-protocol`，防止发布二进制依赖本地 Vite 服务而出现“页面不存在”。
- 新增 Windows 发布验证脚本，检查版本一致性、冻结前端依赖、生产资源、PE/x64 机器类型、哈希和大小。
- 图标生成保持内容不变时不改写文件，避免无意义重链接；安装器构建会验证实际嵌入负载并为最终应用与安装器生成同一份哈希清单。
- 新增 Windows CI：格式、Clippy `-D warnings`、workspace 测试、前端测试/构建、发布二进制和未签名安装器构建。

### 功能与输入

- 补齐 Delete、Insert、Home、End、Page Up/Down、Caps Lock 与 F1–F12。
- 增加最多 256 字的批量文字发送，支持中文、符号和粘贴内容；仍拒绝不安全控制字符。
- 增加独立可靠传输通道：双向纯文本剪贴板和控制端到被控端的文件发送不与输入、控制或视频队列互相阻塞。
- 远程画面中的 `Ctrl+V` 采用显式智能粘贴：先释放控制端残留按键并排空输入队列，再通过加密传输写入远端剪贴板；仅在匹配的远端确认返回后注入粘贴快捷键，不启用后台剪贴板监听。
- 文件接收必须经过被控端原生确认，临时文件校验成功后才原子移动到“下载”文件夹；拒绝、取消和无效内容会清理临时文件，网络断线则保留可验证的断点供用户显式恢复。
- 文件接收暂存区与 Windows“下载”目录位于同一卷，兼容用户把下载目录迁移到其他磁盘；启动新接收前会有界清理超过 24 小时的崩溃遗留分片，且接收决定会明确返回拒绝、占用或本地读写失败。
- 剪贴板与远端文件选择响应必须匹配当前请求 ID；文件邀请会携带其来源请求，迟到或不匹配的响应被忽略。剪贴板、接收确认、远端文件选择、接收停滞、单块发送和最终保存确认均有独立截止时间，超时只结束对应传输，不关闭远程画面与输入会话。
- 控制端用当前 Windows 账户的 DPAPI 加密保存最近一次未完成文件任务，并绑定原目标的完整 128 位设备身份；应用或电脑重启后只允许连接同一设备并由用户显式重试，连接其他电脑时后端拒绝发送。成功保存后立即清理，记录超过 24 小时自动失效，不会在恢复连接后静默重发。WebView 只获得 12 位公开设备 ID、文件名、方向、大小和恢复文案，不接触源文件完整路径。“不再重试”只删除匹配传输 ID 的隐藏断点。断点续传会先验证加密前缀哈希，并在接收前检查下载卷空间，避免同大小源文件变化和磁盘写满造成延迟失败。
- 多文件等待队列同样由 DPAPI 原子保存并绑定完整目标身份；重启或切换连接后只在原设备会话中以暂停状态载入，必须由用户点击“继续队列”。连接其他设备时无法覆盖或误发旧队列，可显式放弃且不会删除源文件。每项在真正发送前重新校验名称、大小和最后修改时间，损坏、过期、重复或越界记录失败关闭。相同状态不会重复写盘或递增 revision；保护失败会立即降级为“仅本次运行”并允许重试，清除失败则恢复内存记录，界面不会把尚未删除的加密记录显示成已清空。队列变更使用一次性后端确认并设有 5 秒上限；断线丢弃的命令、存储错误和超时不会被前端误认为成功，前端以 revision token 串行化操作，迟到完成也不能解锁较新的动作。
- 增加独立有界音频数据报通道：被控端通过 WASAPI 回环采集默认系统输出，转换为 48 kHz 单声道 16 位 PCM；静音时主机停止发送，设备失效时只降级声音而不关闭远控会话。
- 音频队列满时丢弃最旧数据并重置控制端播放调度，优先保持低延迟；音频不会占用输入、控制、文件或视频队列。
- 视频帧调度不再用“距上一帧至少 `1_000_000 / fps`”的严格比较，避免 60 Hz 采集的两个 `16,666 µs` 间隔因少量取整误差无法满足 30 FPS 阈值并退化到约 20 FPS。绝对节奏器带 2 ms 容差、跨过迟到周期且在切屏、画质切换、强制关键帧和时钟回退时重置；颜色转换预计算行列映射并让一个 2×2 采样块同时生成 Y/UV，在本机 release 测量中 1080p 路径由 7.42 ms 降至 4.40 ms。
- 新连接默认启用带迟滞的自动画质，流畅、均衡和清晰档统一保持 30 FPS，仅通过分辨率与码率分层；弱网降档优先保留鼠标与窗口操作的连续感，用户仍可在单次会话中手动锁定档位。
- 自动画质通过 Noise 加密控制通道同时接收网络分片损失与控制端 WebCodecs 队列压力；连续严重积压会保留 Canvas 最后一帧、清空旧解码队列并请求新关键帧，主机据此逐级降档。播放压力按当前视频流绑定并限制数值范围，迟到流、异常 WebView 数值和手动画质都不能改变新会话策略。
- 共享控制端在视频重组完成后检查 H.264 参考帧连续性；确认分片缺失或帧号断层时不再交付后续增量帧，并通过已有加密控制通道请求关键帧。关键帧请求最多每秒重试一次，收到合法关键帧后立即恢复，不会因首次恢复帧再次丢失而永久等待。
- 保持 Windows 安全桌面边界：不宣称支持普通进程无法注入的 `Ctrl+Alt+Delete`。

### 用户体验与可访问性

- 首次运行从“理解并填写 relay/session/key”改成单按钮“启用远程连接”。
- 高级自建中继设置仍可用，但不再阻挡普通用户完成首次配对。
- 删除不可达的局域网网卡选择、专用防火墙提示和旧 relay 状态分支。
- 导航采用 tab 语义、roving tabindex，并支持左右方向键、Home、End。
- 诊断不再把“配置已保存”等同于“网络已连通”，而是根据实际 host runtime 区分连接、重连、等待和停止。
- 控制界面与错误/恢复文案保持中文；配对码的秘密属性和本机审批步骤在关键操作旁明确提示。
- 后台 host 状态事件不会在控制会话中重建整个页面和视频解码器；异步快照只允许最新请求更新界面，避免旧响应覆盖新状态。
- 页面销毁、指针取消和失去捕获时会主动释放远端按键/鼠标状态；批量文字发送期间禁止重复提交。
- 远程全屏改用 Tauri 的 Windows 原生窗口全屏，不再依赖 WebView 页面全屏；窗框和普通导航完全退出布局，工具栏、传输面板与文字输入作为覆盖层保留。适应模式会把画面放大到可用显示区域，并以实际 contain 内容矩形映射鼠标，宽高比黑边不会造成坐标偏移；断线与主动断开都会退出全屏。
- 全屏工具栏在进入后显示 3 秒并自动收起，顶部 12 像素触发区、键盘焦点和已打开的文字/传输面板会固定显示；普通远程输入不会反复唤醒工具栏。主状态只显示分辨率、当前画质和加密状态，帧计数与丢包详情继续进入脱敏诊断。
- 中文输入栏在窄窗口会重排为两列，动态诊断与设备文本统一经过有单测覆盖的 HTML 转义。

- Dependabot 每周检查 Cargo、Windows UI npm 与 GitHub Actions 依赖，降低锁文件长期不更新带来的供应链风险。
- 删除 Tauri 对 relay 服务端、`rcgen`、`rustls`、`quinn` 的重复直接依赖，网络客户端只通过共享 transport crate。
- 控制端只保留设备 ID 与访问密码入口，邀请材料由 Rust 和中继目录内部处理，不再暴露给 WebView。
- 协议 9 为视频网络反馈增加有界播放压力字段；目录继续只比较控制端与主机提交的版本，中继不解析业务控制消息，因此升级只要求两台 Windows 电脑使用同一版本。
- 0.1.58 的参考帧连续性恢复完全位于共享控制端，不修改协议 9 或 relay；新旧协议 9 客户端网络格式兼容，实际分发仍推荐两端使用同一版本。
- 0.1.59 移除大 H.264 帧的 Tauri Channel 推送，改用按视频流与配置版本绑定的串行拉取：WebView2 最多一个请求在途，Rust 最多保留一个待取帧。邮箱溢出后丢弃不安全的增量参考链、最多每秒请求一次关键帧，并把有界播放压力交给自动画质；输入、音频、剪贴板和文件通道不受影响，协议 9 与公网中继保持不变。
- 0.1.60 为串行视频拉取增加最高 2 秒的有界退避，单次 Tauri/WebView2 调用失败不再永久终止画面循环。Rust 邮箱同时记录实际交付、溢出和关键帧替换，控制端记录拉取失败；这些脱敏累计值进入关联会话分析，用于区分网络已完成帧、本机 IPC 停滞与前端解码显示问题。协议 9、中继和业务密文均不改变。
- 0.1.61 优化控制端高频路径：状态信号按动画帧合并，输入队列采用无搬移头指针，远程游标缓存节点，Canvas 开启低延迟绘制并对同尺寸视频帧走直接绘制。视频拉取边界、协议 9 和公网中继均不改变。
- 0.1.62 加固会话生命周期：主机 supervisor 发生不可恢复错误时发布明确的停止事件；控制端后端已停止后拒绝迟到命令，避免旧任务继续接收输入或传输操作。协议 9 和公网中继均不改变。
- 0.1.63 扩展本地脱敏诊断导出：合并主机与控制端日志并按时间排序、标注来源，避免控制端测试证据只留在独立日志文件中。协议 9、中继和业务数据均不改变。
- 0.1.64 收敛连接状态更新的渲染边界：稳定连接仅就地更新状态徽标，只有连接状态或视频流身份变化才重建远程表面，避免重置 WebCodecs 解码器。协议 9、中继和业务数据均不改变。
- 0.1.65 优化瞬时重连体验：若已有远程视频表面，重连等待期间不清空解码器和最后一帧，只显示恢复覆盖层；真正恢复或停止时再按生命周期重建。协议 9、中继和业务数据均不改变。
- 0.1.66 将远程画面保留策略抽为纯生命周期状态机，覆盖“已连接 → 重连 → 建立连接 → 等待确认 → 恢复/停止”路径，避免状态文案更新误触发解码器重建。协议 9、中继和业务数据均不改变。

- 0.1.68 在本地诊断导出中根据最近的中继、解码和显示指标生成脱敏性能摘要，明确下一步排查方向，不改变默认界面和运行时开销。协议 9、中继和业务数据均不改变。
- 0.1.69 为性能摘要增加时间窗口，避免把旧会话的中继指标与当前渲染指标混合，诊断不足时明确提示重新复现。协议 9、中继和业务数据均不改变。

## 测试策略

### 每次提交

- TypeScript：设备凭据、产品配置、输入映射、音频包解析、键盘导航、并发请求门禁和动态文本转义单测与生产构建。

- 发布：版本一致性、生产资源检查、启用 `custom-protocol` 的 x64 PE 构建、哈希清单、安装器构建。

### Windows 硬件门禁

- 真实 DXGI 多显示器采集、屏幕切换与 Media Foundation H.264 编码。
- WASAPI 默认输出回环、静音/恢复、默认设备切换，以及播放声音时的鼠标与画面流畅度。
- relay 连续故障后的全 runtime 重建。
- suspend/resume callback 注册与去重。
- Noise 加密媒体、光标和输入持续传输。

### 发布前人工矩阵

- Windows 10/11，至少 Intel 与 AMD/NVIDIA 两类图形环境。
- 两台物理电脑：同网、跨网、50–80ms RTT、短时丢包、断网、Wi-Fi 切换、实际睡眠/唤醒。
- 新装、覆盖升级、登录启动、普通卸载、`--remove-data` 卸载。
- 已签名安装包的 Authenticode、RFC 3161 时间戳和 SmartScreen。

## 未自动解决的发布风险

这些事项需要证书、基础设施或物理设备，不能仅靠仓库代码完成：

1. 购买/配置公有代码签名证书或 Microsoft Artifact Signing，并积累 SmartScreen 信誉。
2. 公网 relay 已有外部 TLS/QUIC 握手监控；仍需增加第二节点、容量告警、证书续期失败告警和故障切换演练。
3. 在两台真实 Windows 电脑上完成数小时长稳和实际电源状态验收。
4. 已实现只接受 GitHub 稳定 Release、匹配 x64 安装器和已签名清单的版本检查；在代码签名身份就绪前不创建正式 Release，也不启用后台下载或自动安装。
5. 继续验证多显示器在不同 DPI 与排列下的坐标一致性；依据真实使用反馈决定远程音频和自动更新的产品优先级，避免在没有使用证据时扩大攻击面。
