# DeskLink Windows 架构复盘

- 日期：2026-07-18
- 范围：Windows 10/11 x64 桌面端、共享 Rust 核心、QUIC relay、安装器与发布门禁。

## 结论

DeskLink 的安全会话与 Windows 媒体链路已经具备可靠的 MVP 基础，主要问题不在“Rust/Tauri 是否能用”，而在发布形态与产品默认路径曾经同时存在两套架构：现代 Tauri 应用旁仍打包旧辅助 host，公网中继旁仍保留内置局域网中继，源码默认值、文档和已构建产物也发生过漂移。

本轮将 Windows 产品收敛为一个入口、一套网络模型和一条发布链：

```text
DeskLink.exe
├─ WebView2 UI：中文界面、WebCodecs、受限输入采集
└─ Rust/Win32 安全边界
   ├─ DPAPI 身份 / 配置 / 信任
   ├─ 本地原生批准与撤销
   ├─ QUIC + 系统 TLS 校验
   ├─ Noise 双向认证与业务 AEAD
   ├─ DXGI → Media Foundation H.264
   └─ 有界 SendInput
          │
          ▼
公网或自建 relay：只匹配会话并转发业务密文
```

## 已解决的问题

### 网络与安全

- 删除桌面进程内的临时自签名局域网 relay，以及跳过服务器证书身份校验的客户端路径。
- 默认使用部署中的公网 relay，并通过系统信任链校验 `turn.p2p.yxswy.com`。
- 首次运行由 Rust 生成会话 ID 与 relay 凭据，DPAPI 加密保存；WebView 不接触已保存密钥。
- 删除旧版连接码入口与 loopback 配置迁移，未发布版本统一使用设备 ID、访问密码和公网中继目录。
- 普通常驻主机在单轮重试预算耗尽后会按最大退避继续下一轮恢复，不会因短时会话占用永久停机；一次性配对仍严格受邀请过期时间约束。
- 协议层拒绝 F13+、NUL/控制字符、越界指针/滚轮和未知修饰键，不把合法性只留给最终 `SendInput`。

### 发布架构

- 安装器只部署 `DeskLink.exe`；升级时清理旧 `desklink-windows.exe`，避免两个 host 生命周期并存。
- Tauri 发布构建固定启用 `custom-protocol`，防止发布二进制依赖本地 Vite 服务而出现“页面不存在”。
- 新增 Windows 发布验证脚本，检查版本一致性、冻结前端依赖、生产资源、PE/x64 机器类型、哈希和大小。
- 图标生成保持内容不变时不改写文件，避免无意义重链接；安装器构建会验证实际嵌入负载并为最终应用与安装器生成同一份哈希清单。
- 新增 Windows CI：格式、Clippy `-D warnings`、workspace 测试、前端测试/构建、发布二进制和未签名安装器构建。

### 功能与输入

- 补齐 Delete、Insert、Home、End、Page Up/Down、Caps Lock 与 F1–F12。
- 增加最多 256 字的批量文字发送，支持中文、符号和粘贴内容；仍拒绝不安全控制字符。
- 保持 Windows 安全桌面边界：不宣称支持普通进程无法注入的 `Ctrl+Alt+Delete`。

### 用户体验与可访问性

- 首次运行从“理解并填写 relay/session/key”改成单按钮“启用远程连接”。
- 高级自建中继设置仍可用，但不再阻挡普通用户完成首次配对。
- 删除不可达的局域网网卡选择、专用防火墙提示和旧 relay 状态分支。
- 导航采用 tab 语义、roving tabindex，并支持左右方向键、Home、End。
- 诊断不再把“配置已保存”等同于“网络已连通”，而是根据实际 host runtime 区分连接、重连、等待和停止。
- 控制界面与错误/恢复文案保持中文；配对码的秘密属性和本机审批步骤在关键操作旁明确提示。
- 后台 host 状态事件不会在控制会话中重建整个页面和视频解码器；异步快照只允许最新请求更新界面，避免旧响应覆盖新状态。
- 页面销毁、指针取消和失去捕获时会主动释放远端按键/鼠标状态；批量文字发送期间禁止重复提交。
- 中文输入栏在窄窗口会重排为两列，动态诊断与设备文本统一经过有单测覆盖的 HTML 转义。

### 可维护性与测试

- 浏览器按键、修饰键、鼠标和滚轮映射提取为纯模块并增加单元测试。
- 公网产品配置与 tab 键盘导航提取为纯模块并增加单元测试。
- 发布门禁检查 Rust 与 TypeScript 的正式中继地址一致，并扫描全部生产 HTML/CSS/JS 资源中的开发服务器 URL。
- 新增使用生产 transport 客户端的公网 TLS/QUIC 探针，以及每半小时执行的 Windows GitHub Actions 中继监控。
- Dependabot 每周检查 Cargo、Windows UI npm 与 GitHub Actions 依赖，降低锁文件长期不更新带来的供应链风险。
- 删除 Tauri 对 relay 服务端、`rcgen`、`rustls`、`quinn` 的重复直接依赖，网络客户端只通过共享 transport crate。
- 控制端只保留设备 ID 与访问密码入口，邀请材料由 Rust 和中继目录内部处理，不再暴露给 WebView。

## 测试策略

### 每次提交

- TypeScript：设备凭据、产品配置、输入映射、键盘导航、并发请求门禁和动态文本转义单测与生产构建。
- Rust：workspace 单元/集成/文档测试；格式；全 target/all feature Clippy 且警告视为错误。
- 发布：版本一致性、生产资源检查、启用 `custom-protocol` 的 x64 PE 构建、哈希清单、安装器构建。

### Windows 硬件门禁

- 真实 DXGI 多显示器采集、屏幕切换与 Media Foundation H.264 编码。
- relay 连续故障后的全 runtime 重建。
- suspend/resume callback 注册与去重。
- Noise 加密媒体、光标和输入持续传输。

### 发布前人工矩阵

- Windows 10/11，至少 Intel 与 AMD/NVIDIA 两类图形环境。
- 两台物理电脑：同网、跨网、50–80ms RTT、短时丢包、断网、Wi-Fi 切换、实际睡眠/唤醒。
- 新装、覆盖升级、登录启动、普通卸载、`--remove-data` 卸载。
- 已签名安装包的 Authenticode、RFC 3161 时间戳和 SmartScreen。

## 未自动解决的发布风险

这些事项需要证书、基础设施或物理设备，不能仅靠仓库代码完成：

1. 购买/配置公有代码签名证书或 Microsoft Artifact Signing，并积累 SmartScreen 信誉。
2. 公网 relay 已有外部 TLS/QUIC 握手监控；仍需增加第二节点、容量告警、证书续期失败告警和故障切换演练。
3. 在两台真实 Windows 电脑上完成数小时长稳和实际电源状态验收。
4. 选择并实现可信的自动更新通道；在此之前版本升级继续使用签名安装器。
5. 继续验证多显示器在不同 DPI 与排列下的坐标一致性；依据用户验证决定剪贴板、文件传输和音频的产品优先级，避免在没有使用证据时扩大攻击面。
