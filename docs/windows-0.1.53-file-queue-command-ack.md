# Windows 0.1.53：等待队列操作确认与并发互斥

0.1.53 解决文件队列按钮过去只等待“命令进入 Rust 通道”、没有等待状态机真正处理的问题。该差异在连接切换、磁盘错误或连续点击时会造成按钮过早恢复，甚至让用户误以为操作已经完成。

## 后端确认

- 移除文件、清空等待、继续队列和重试保护均携带独立的一次性确认通道；
- Tauri 命令只有在连接状态机修改队列并完成本地保护处理后才返回；
- DPAPI 写入或清除失败会把原始安全中文原因返回给调用方，同时仍发布真实的 `memoryOnly` 状态；
- 会话在处理前切换或结束时，确认发送端随命令释放，调用方立即得到“会话已经变化”；
- 单次等待最多 5 秒，防止状态机异常时界面永久锁定；陈旧移除和重复继续会被后端拒绝，而不是静默成功。

## 前端互斥

- 文件选择、拖放、恢复记录清理和四种队列变更共享忙碌边界；
- 后端确认返回前，其他队列按钮保持禁用，并显示“正在移除”“正在清空”“正在继续”或“正在重试”；
- 每个操作具有递增 token，迟到的 `finally` 只能结束自身，不能清除更新操作的忙碌状态；
- 所有新增图标继续使用 Lucide，不引入手写 SVG。

## 协议与部署

本阶段只调整 Windows Tauri 进程内部命令确认和 WebView 状态，不改变 DeskLink 网络协议。协议版本保持 8，公网中继无需重新部署。
