# Windows 4K 局域网实验档门槛

这份实验档只定义“何时可以开始测量”，不把 4K 伪装成当前稳定功能。默认公网中继仍使用 2560×1440。

## 当前测量基线

| 模式 | 码率 | 帧率 | 平均编码帧 | Datagram 分片 | NV12 原始帧 |
| --- | ---: | ---: | ---: | ---: | ---: |
| 清晰档 | 18 Mbps | 30 FPS | 75,000 B | 74 | 5,529,600 B |
| 4K 实验档 | 40.5 Mbps | 30 FPS | 168,750 B | 165 | 12,441,600 B |

这些数值由 `desklink_protocol::VideoFrameBudget` 计算。它们是平均值，突发场景可能更高；当前 1024 字节 Datagram 和 4096 分片上限可以容纳平均帧，但不代表丢包后的完整帧恢复已经足够可靠。

## 必须同时满足的门槛

1. 传输路径明确标记为 Direct LAN；当前 Windows 运行时已经有同局域网候选、认证探测和失败回落，但还没有经过两台物理电脑验收，因此实验档仍没有用户入口。
2. 主机和控制端都显式声明 3840×2160 能力，并且控制端解码器支持 High Profile。
3. 连接稳定窗口内 RTT、丢包率和解码队列均低于实验阈值；任何一项连续恶化都要自动回到 2560×1440。
4. 4K 编码器初始化、首个关键帧、连续 30 帧和强制关键帧恢复全部成功。
5. 诊断同时记录实际尺寸、Profile、分片数、丢弃帧和回退原因；不能只依据“编码器初始化成功”判定可用。

传输层现在已经有 `VideoPathKind` / `VideoPathQuality` / `VideoPathDecision` 门控：`Relay` 永远不能进入 4K，未来只有 `DirectLan` 且 RTT、丢包同时在实验预算内时才允许继续。当前探测只测得 RTT，丢包仍标记为未知，因此状态机不会因一次握手成功就放行 4K。控制端诊断同时记录 `video_width`、`video_height`、实际 Profile 和 `video_path`，便于后续用真实数据判断是否值得开放 4K。

## 实验实现顺序

1. 先完成 Direct LAN 运行时双机验收，并把路径类型绑定到加密会话 transcript；公网 NAT 穿透仍是后续独立项目。
2. 在能力协商中加入实验尺寸和帧预算，不修改默认 MVP 上限。
3. 加入关键帧优先、分片预算和丢包回退策略；在恢复策略完成前不打开公网 4K。
4. 用 `experimental_4k_media_foundation_encoder_initializes` 探针确认具体 Windows 硬件 MFT 的 4K 上限。
5. 最后才在 UI 中提供“局域网实验”入口，并明确显示实验状态。
